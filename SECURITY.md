## 2. AI Code Audit & Attribution

**AI Tool Used:** Claude (Anthropic)

**The Veto — Instance #1:**
During initial prototyping, the AI suggested building the search feature using raw string interpolation in a SQL query:

```python
# AI's original suggestion (REJECTED):
results = conn.execute(f"SELECT * FROM cves WHERE description LIKE '%{query}%'")
```

This is a textbook SQL injection vulnerability. Any user input containing `'` characters or SQL metacharacters would corrupt or escape the query. The corrected implementation uses parameterized queries exclusively:

```python
# Secure implementation (ACCEPTED):
rows = conn.execute(
    "SELECT * FROM cves WHERE description LIKE ? LIMIT ?",
    (f"%{query}%", limit)
)
```

**The Veto — Instance #2:**
The AI initially suggested rendering CVE descriptions using `innerHTML` on the frontend for convenience. This was rejected because CVE descriptions occasionally contain characters that browsers interpret as HTML, creating a vector for reflected XSS. All dynamic content is now inserted via `textContent` or explicit DOM construction:

```javascript
// AI's suggestion (REJECTED):
descEl.innerHTML = data.description;

// Secure implementation (ACCEPTED):
descEl.textContent = data.description;
```

## 3. Hardening Measures (The "Big Two")

| Feature | Implementation Detail | Threat Mitigated |
|---|---|---|
| **Input Validation & Sanitization** | All CVE IDs validated against strict regex `^CVE-\d{4}-\d{4,}$` before any processing. All user input passed through `bleach.clean()` to strip HTML/JS. Frontend uses `textContent` exclusively for user-derived data — never `innerHTML`. No `eval()` or `dangerouslySetInnerHTML` anywhere in the codebase. | XSS, Injection, Parameter Tampering |
| **Rate Limiting** | Flask-Limiter enforces per-endpoint rate limits: CVE lookups (40/min), search queries (30/min), forced refresh (5/min), NVD sync trigger (2/hr). Returns HTTP 429 with a descriptive error on breach. Prevents scraping, DoS, and API abuse. | Denial of Service, Brute Force, API Abuse |
| **Security Response Headers** | Every response includes: `Content-Security-Policy` (restricts script/style/connect sources), `X-Frame-Options: DENY` (clickjacking), `X-Content-Type-Options: nosniff` (MIME sniffing), `X-XSS-Protection`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy`. | Clickjacking, MIME Sniffing, XSS, Information Leakage |
| **Parameterized SQL Queries** | Zero string interpolation in any SQL statement throughout `database.py`. Every query uses `?` placeholders with separate value tuples. SQLite FTS5 used for full-text search with proper escaping. | SQL Injection |

## 4. OWASP Top 10 Risk Assessment

| OWASP Risk | Mitigation Applied |
|---|---|
| A01 – Broken Access Control | Rate limiting on all mutation endpoints; no authentication bypass possible |
| A03 – Injection | Parameterized SQL queries; strict CVE ID regex whitelist; bleach sanitization on all input |
| A05 – Security Misconfiguration | Security headers on all responses; `debug=False` enforced; `SECRET_KEY` from env |
| A07 – Identification & Auth Failures | Rate limiting prevents brute force; no plaintext credentials stored |
| A09 – Security Logging | All requests logged; update events logged to `update_log` table |
